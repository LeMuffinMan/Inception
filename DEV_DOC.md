# DEV_DOC — Developer Documentation

## Setting up the environment from scratch

### Prerequisites

The following must be installed on your virtual machine:

- **Docker**
- **Docker Compose**
- **make**
- **git**
- **openssl** (used by the setup scripts to generate secrets, or it will use `/dev/urandom` as fallback)
- **curl** (used by the check script for HTTPS and TLS probes)
- **lftp** or **ftp** (used by `ftp.sh` for FTP operations)

> The project must be run inside a virtual machine with a sudo-privileged user. Some commands (notably `make fclean`) require `sudo` to remove volume data from the host filesystem.

### Scripts reference

All scripts source `scripts/lib/config.sh` and `scripts/lib/format.sh` at the top.

#### `lib/config.sh`
The single configuration file for the entire project. Contains:
- `LOGIN` / `DOMAIN` — derived from `$(whoami)` by default
- `CREDENTIALS_FILES` — list of secret files to generate
- `REQUIRED_ENV_VARS` — list of `.env` variables to check and fill interactively
- Container names, expected ports, TLS versions to accept/reject, volume names, timeouts
- `COMPOSE_FILE`, `ENV_FILE`, `SECRETS_DIR` — path resolution

**This is the only file you need to edit to adapt the project to a different setup.**

#### `lib/format.sh`
Shared formatting library. Provides:
- `log_info / log_debug / log_warn / log_error` — colored output with level prefix
- `check <label> ok|ko [detail]` — displays a pass/fail line with aligned columns
- `skip <label>` — displays a "skipped (already exists)" line
- `section <title>` — draws a bordered section header
- `header <title> <subtitle>` — prints a top-level header
- `wait_for <timeout> <command>` — retries a command until it returns 0 or timeout
- `wait_for_containers` — polls all containers in `CONTAINERS_TO_TEST` until they are `healthy`

#### `generate_secrets.sh`
Called by `make up` and `make generate-secrets`. Two phases:

1. **`.env` phase** — if `srcs/.env` is missing or empty, prompts to auto-generate it with defaults via `auto_generate_env.sh`. Then sources `.env` and loops over `REQUIRED_ENV_VARS`: any missing variable is filled interactively. Special case: `MYSQL_ADMIN_EMAIL` is checked against `MYSQL_USER_EMAIL` to prevent duplicates (WordPress requires two distinct email addresses).
2. **Secrets phase** — creates `secrets/` if absent, then loops over `CREDENTIALS_FILES`. Existing files are skipped. New files are written with `openssl rand -base64` (or `/dev/urandom` fallback). All files are `chmod 600`. The `-f` flag forces regeneration (with a confirmation prompt before deleting `secrets/`).

#### `auto_generate_env.sh`
Writes `srcs/.env` with nine variables, all derived from `$USER`:
```
MYSQL_DATABASE, MYSQL_USER, MYSQL_USER_EMAIL, MYSQL_ADMIN_EMAIL,
DOMAIN_NAME, WP_TITLE, WP_USER, WP_ADMIN_USER, FTP_USER
```

#### `edit_hosts.sh`
Called by `make up` and `make edit-host`. Two actions:

1. **`/etc/hosts`** — if the domain is not already present, prompts `y/n` to add it. If `127.0.0.1 localhost` is found, comments it out and appends the domain below. If not found, inserts the domain at the very top of the file as a fallback.
2. **`vm.overcommit_memory`** — if currently set to `0`, prompts `y/n` to enable it (`sysctl vm.overcommit_memory=1`). Required for Redis to work correctly (see config note in the file). Setting is not persistent across reboots — to make it permanent, add `vm.overcommit_memory=1` to `/etc/sysctl.conf`.

#### `check_inception.sh`
Called by `make check` and `make check-<service>`. Accepts an optional service name argument to restrict checks to a single section.

Sections and what they verify:

| Section | Checks |
|---|---|
| **MariaDB** | running, healthy, log (`ready for connections`), `SELECT 1` probe, port `3306/tcp`, volume |
| **NGINX** | running, healthy, logs, HTTPS probe, TLS version detected, TLSv1.2 and 1.3 accepted, TLSv1.1 rejected, port `0.0.0.0:443` |
| **WordPress** | running, healthy, log (`WP installed`), port `9000/tcp`, volume, ≥2 users in DB, admin exists, admin username matches no forbidden pattern |
| **Redis** | running, healthy, `redis-cli PING`, cache populated (keys > 0), `wp redis status` via WP-CLI, port 6379 |
| **Adminer** | running, healthy, HTTP probe on port 8080, reachable via NGINX at `/adminer`, port not exposed on host, connects to MariaDB |
| **vsftpd** | FTP list (verifies `wp-config.php` visible), FTP download, FTP upload |
| **Project Integrity** | no password in Dockerfiles, `.env` exists with all required vars, no `latest` tag in compose, all `FROM` use Alpine or Debian with a pinned tag, no external images in compose |
| **Network** | NGINX is the only container with a host-exposed port, no port 80, no `links:` in compose, no `network: host`, domain reachable, Hugo and ChessGame reachable, `/etc/hosts` entry present |

#### `crash_test.sh`
Called by `make crash`. Runs two passes:
1. **Individual crash** — for each container in `CONTAINERS_TO_TEST`: finds its PID via `docker inspect`, sends `kill -9`, then waits up to `RESTART_TIMEOUT` seconds for the container to come back healthy.
2. **Simultaneous crash** — kills all container PIDs at once, then checks each one recovers.

#### `ftp.sh`
Called by `make ftp-list`, `make ftp-dl-*`, `make ftp-up-*`. Reads credentials from `secrets/ftp_pass.txt` and `FTP_USER` from `srcs/.env`. Uses the system `ftp` client with a heredoc to issue commands non-interactively.

Flags: `-l` (list), `-d <file>` (download), `-u <file>` (upload).

#### `kill_containers.sh`
Called by `make kill` (used internally by `make re` and `make fclean`). Iterates over `CONTAINERS_TO_TEST` and runs `docker kill` on each running container.

#### `check_sudo.sh`
Called before any target that requires `sudo`. Runs `sudo true` silently — exits with an error and a clear message if privileges are unavailable.

### Automated setup (recommended)

After cloning the repository inside your VM:

```bash
git clone https://github.com/LeMuffinMan/Inception
cd Inception && make
```

`make` handles the entire setup automatically:

1. Prompts to generate `srcs/.env` with `$USER`-based defaults, then fills missing variables interactively
2. Generates missing password files in `secrets/` using `openssl rand` (existing files are skipped)
3. Prompts to edit `/etc/hosts` and enable `vm.overcommit_memory` for Redis
4. Creates `~/data/mysql`, `~/data/wordpress`, `~/data/hugo`, and `~/data/chessgame` on the host
5. Builds all Docker images
6. Starts all containers in detached mode
7. Runs the full validation script

The WordPress site will be reachable at `https://<login>.42.fr`.

### Manual setup (custom deployment)

If you prefer not to use the automated scripts, provide the following manually.

#### 1. Secret files

Create the `secrets/` folder at the root of the repository and populate it with the following files (one value per file, no trailing newline):

| File | Contents |
|---|---|
| `secrets/db_password.txt` | MariaDB application user password |
| `secrets/db_root_password.txt` | MariaDB root user password |
| `secrets/wp_admin_password.txt` | WordPress admin password |
| `secrets/wp_user_password.txt` | WordPress regular user password |
| `secrets/ftp_pass.txt` | FTP user password |

#### 2. Environment file

Create `srcs/.env`:

```env
DOMAIN_NAME=<login>.42.fr
MYSQL_DATABASE=<login>_db
MYSQL_USER=<login>_mysql_user
MYSQL_USER_EMAIL=<login>@mail.fr
MYSQL_ADMIN_EMAIL=<login>_su@mail.fr
WP_TITLE=<login>_wordpress
WP_USER=<login>_wordpress_user
WP_ADMIN_USER=<login>_wordpress_su
FTP_USER=<login>
```

#### 3. Data directories

```bash
mkdir -p ~/data/mysql ~/data/wordpress ~/data/hugo ~/data/chessgame
```

#### 4. `/etc/hosts`

```
127.0.0.1   <login>.42.fr
```

#### 5. Redis memory overcommit

```bash
sudo sysctl vm.overcommit_memory=1
```

### Custom deployment for a new user

The scripts are designed to be fully portable across local users. To deploy as a new user:

```bash
sudo useradd -m -s /bin/bash <new_user>
sudo usermod -aG sudo <new_user>
sudo usermod -aG docker <new_user>
passwd <new_user>
su - <new_user>
```

Then clone and run:

```bash
git clone https://github.com/LeMuffinMan/Inception
cd Inception && make
```

All credentials and login-dependent variables are derived from the current username automatically.
To adjust defaults, edit `scripts/lib/config.sh`.

## Building and launching with Make and Docker Compose

### Full command reference

| Command | Effect |
|---|---|
| `make` / `make all` | Generate secrets and `.env`, edit hosts, create data dirs, build images, start containers, run full check |
| `make up` | Same as above without running the check script |
| `make down` | Stop and remove containers; preserve volumes and data |
| `make re` | Kill containers, wipe volumes, rebuild everything from scratch |
| `make restart-<service>` | Restart a specific container (e.g. `make restart-nginx`) |
| `make status` | Show running containers and volumes |
| `make build-<service>` | Rebuild a single service image from scratch (e.g. `make build-wordpress`) |
| `make logs` | Stream live logs from all containers |
| `make logs-<service>` | Stream logs of a specific service in a new terminal window |
| `make shell-<service>` | Open a shell inside a specific container in a new terminal window |
| `make check` | Run the full validation script |
| `make check-<service>` | Run validation for a specific service only |
| `make crash` | Kill each container's PID and verify automatic restart |
| `make ftp-list` | List files on the FTP server |
| `make ftp-dl-<file>` | Download a file from the FTP server |
| `make ftp-up-<file>` | Upload a file to the FTP server |
| `make regenerate-secrets` | Force-regenerate all password files (overwrites existing, with confirmation) |
| `make edit-host` | Re-run the `/etc/hosts` and overcommit setup |
| `make clean` | Stop and remove containers (alias for `make down`) |
| `make fclean` | Full cleanup: containers, volumes, images, build cache, `~/data`. **Requires `sudo`** |
| `make uninstall` | `fclean` + restore `/etc/hosts` + prompt to delete `secrets/` and `srcs/.env` |
| `make reinstall` | `uninstall` then regenerate secrets, edit hosts, rebuild everything |

## Data persistence

Both named volumes store data on the host machine under `~/data`:

| Volume | Host path | Contents |
|---|---|---|
| `srcs_mariadb_data` | `~/data/mysql` | MariaDB database files |
| `srcs_wordpress_data` | `~/data/wordpress` | WordPress core files and uploads |

These directories survive container stops, restarts, and rebuilds. They are only removed by `make fclean`.

The named volume configuration in `docker-compose.yml` uses `driver_opts` to bind to the host paths while remaining proper named volumes (not raw bind mounts, which are forbidden by the subject):

```yaml
volumes:
  db_volume:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: /home/<login>/data/mysql
  wp_volume:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: /home/<login>/data/wordpress
```

## Key architectural constraints

- No container uses `network: host`, `--link`, or `links:`.
- No container is started with infinite-loop commands (`tail -f`, `sleep infinity`, `while true`, `bash`, etc.).
- Containers run foreground processes as PID 1 — daemons are invoked directly, not wrapped in looping scripts.
- No passwords are hardcoded in Dockerfiles — all sensitive values are injected via Docker secrets.
- The `latest` tag is forbidden for all images.
- Images are built from the penultimate stable version of Alpine or Debian.
- NGINX is the **only** container exposed to the outside, on port **443** only, using **TLSv1.2 or TLSv1.3**.
- The WordPress database contains two users: one administrator (username must not contain `admin` or `administrator`) and one regular user.
